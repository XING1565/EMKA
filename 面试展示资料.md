# EMKA 面试展示资料

## 1. 项目一句话介绍

EMKA 是一个面向企业知识场景的多模态知识 Agent 工作区。用户可以上传文档、表格、图片等资料，系统会完成解析、分块、索引、RAG 检索、工具调用、记忆读写和执行轨迹记录，最终在中文工作台中返回回答、报告、引用证据和可观测运行过程。

适合在面试中这样介绍：

> 这是一个企业多模态知识 Agent Demo。它不是单纯的聊天 UI，而是把“资料摄取、知识检索、Agent 编排、工具调用、记忆管理、Trace 可观测”串成了一条完整链路。

## 2. 项目解决的问题

企业内部知识通常分散在文档、表格、图片、会议纪要和报告中，普通聊天机器人很难回答“基于这些资料”的问题，也很难说明回答依据来自哪里。

EMKA 解决的核心问题：

- 把多种格式的资料统一摄取为可检索知识。
- 通过 RAG 检索找到与问题相关的内容片段。
- 通过 Agent Runtime 自动完成路由、规划和工具调用。
- 返回回答的同时展示引用证据，降低黑盒感。
- 记录 Trace、工具调用、记忆操作，方便调试和解释系统行为。

## 3. 技术栈

| 模块 | 技术 | 作用 |
| --- | --- | --- |
| 前端 | React、Vite、TypeScript | 构建中文工作台界面 |
| UI 图标 | lucide-react | 提供统一图标组件 |
| 后端 | FastAPI、Pydantic | 提供 API 和运行时模型 |
| ORM | SQLAlchemy | 管理数据库访问 |
| 关系型数据库 | PostgreSQL | 存储文档、分块、记忆、Trace、工具调用 |
| 向量数据库 | Milvus | 存储文档 chunk 向量并做相似度检索 |
| 文档解析 | PyPDF2、python-docx、openpyxl、Pillow | 解析 PDF、docx、xlsx、图片元数据 |
| 容器化 | Docker Compose | 一键启动前端、后端、数据库和向量库 |
| 测试 | pytest、httpx | 单元测试和集成测试 |

## 4. 系统架构

```text
React 中文工作台
  -> FastAPI API
    -> Documents API
      -> 多模态摄取
      -> 文档解析 / 归一化 / 分块 / embedding / 向量索引
      -> PostgreSQL + Milvus

    -> Chat API
      -> Agent Runtime
      -> Memory Service
      -> Intent Router
      -> Task Planner
      -> Tool Registry
      -> RAG Retriever
      -> Trace Recorder

    -> Trace API / Memories API
      -> 前端洞察面板展示执行过程
```

面试讲解重点：

- 前端不直接处理知识逻辑，只负责上传、提问和展示结果。
- 后端将“文档摄取”和“聊天运行时”拆成两条主链路。
- PostgreSQL 存业务数据，Milvus 存向量数据。
- Trace 是一等能力，用于解释 Agent 为什么这样回答。

## 5. 核心功能

### 5.1 多模态资料上传

支持文件：

- 文本：`txt`、`md`、`pdf`、`docx`
- 表格：`csv`、`xlsx`
- 图片：`png`、`jpg`、`jpeg`

处理流程：

```text
上传文件
  -> 判断模态
  -> 选择解析器
  -> 内容归一化
  -> 写入文档表
  -> 文档分块
  -> 生成 embedding
  -> 写入向量库
```

### 5.2 RAG 检索与引用证据

用户提问后，系统会从已索引资料中检索相关 chunk，并返回：

- 文档标题
- 内容片段
- 相关度分数
- 引用字段
- 表格行范围
- 图片 OCR 置信度字段

这使得回答可以追溯来源，而不是只有一段不可解释的文本。

### 5.3 Agent Runtime

聊天请求进入运行时后，会经过：

```text
读取记忆
  -> 意图路由
  -> 任务规划
  -> 工具调用
  -> RAG 检索
  -> 生成回答和报告
  -> 写入记忆
  -> 记录 Trace
```

当前已实现的内部工具：

| 工具 | 作用 |
| --- | --- |
| `search_docs` | 检索知识库文档 |
| `read_doc` | 读取指定文档内容 |
| `summarize` | 基于上下文生成摘要 |
| `generate_report` | 生成 Markdown 报告 |

### 5.4 记忆管理

项目支持三类记忆：

- `session`：当前会话记忆
- `user`：用户级记忆
- `knowledge`：知识相关记忆

聊天开始时读取记忆，结束时写入本轮 message 和 answer，并在 Trace 中记录记忆操作。

### 5.5 Trace 可观测性

每次聊天都会记录：

- 用户问题
- 意图和置信度
- 路由结果
- 任务计划
- 检索文档
- 工具调用
- 记忆读写
- 状态、错误和耗时

这部分是项目的技术亮点之一，可以向面试官强调：Agent 系统难点不只是回答生成，而是过程可解释、可调试、可追踪。

## 6. 前端展示亮点

新版界面已经改成中文 AI 工作台风格：

- 左侧：知识库、工具、文件上传、本地文档库、最近对话。
- 中间：沉浸式对话区、快捷提示词、底部输入框。
- 右侧：运行与证据，包括执行轨迹、检索文档、工具调用、记忆、报告和提取操作。

交互优化：

- 支持 `Enter` 发送，`Shift+Enter` 换行。
- 顶部模式按钮可切换：智能编排、搜索知识库、自动调用工具。
- 左侧知识库和工具可切换。
- 右侧洞察面板可收起。
- 所有用户可见文案已中文化。

## 8. 面试官常见问题与回答

### Q1：为什么使用 PostgreSQL + Milvus？

PostgreSQL 适合存结构化业务数据，例如文档、分块、记忆、Trace、工具调用；Milvus 适合存向量并做相似度检索。检索时先在 Milvus 中找到相似 chunk，再回 PostgreSQL 取文档标题、片段、模态和引用字段。

### Q2：这个项目是否调用真实大模型？

当前没有。项目定位是稳定可复现的 Demo，因此 LLM、embedding 和 OCR 都使用 mock 或 deterministic 实现。这样不依赖外部 API Key、网络和模型延迟。生产化时可以替换为真实 LLM、embedding provider 和 OCR 服务。

### Q3：项目最有技术含量的地方是什么？

不是单个聊天框，而是完整链路：

- 多模态资料摄取
- RAG 索引和检索
- Agent 路由和任务规划
- 工具注册与调用
- 记忆读写
- Trace 可观测

这些模块串起来后，形成了一个可解释、可调试的企业知识 Agent 原型。

### Q4：如果要上线生产，还需要补什么？

需要补齐：

- 真实 LLM、embedding 和 OCR 接入
- 用户登录、鉴权、租户隔离
- 异步任务队列，处理大文件上传和解析
- 监控、日志、告警和成本追踪
- 检索质量评估、引用准确率评估
- 数据安全、备份恢复和权限审计

### Q5：如何保证回答可信？

系统返回回答的同时会返回 retrieved docs，每条证据包含标题、片段、相关度和引用字段。Trace 还会记录意图、计划、工具调用和记忆操作。用户可以追溯回答依据，开发者也可以定位问题。

### Q6：多模态能力体现在哪里？

上传阶段支持文本、表格和图片。文本会被解析为内容块；表格会保留行范围等元数据；图片会读取格式、尺寸并生成 OCR 相关字段。检索结果中保留 modality，前端可以展示不同来源的证据。

## 9. 当前边界

需要主动说明，避免面试官误解：

- 当前 LLM 是 mock 行为，不代表真实模型生成质量。
- 当前 OCR 文本是 mock 输出，但会真实读取图片元数据。
- 未配置 Milvus 时可回退到内存向量库，适合本地开发和测试。
- 当前没有用户登录、鉴权、权限隔离。
- 项目定位是 Demo workspace，不是生产部署模板。
