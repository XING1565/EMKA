EMKA 面试演示资料：企业多模态知识 Agent

一、项目背景

企业内部知识通常分散在项目说明、会议纪要、指标表格、截图和历史报告中。普通问答系统只能处理单一文本输入，很难把多种资料统一索引，也很难解释回答来源。EMKA 的目标是构建一个可运行的企业知识 Agent 工作区，让用户上传资料后，可以通过中文界面完成问答、摘要、报告生成和证据追踪。

二、核心业务痛点

1. 知识入口分散：文档、表格、图片和报告格式不同，人工整理成本高。
2. 回答缺少依据：用户希望知道回答引用了哪些资料，而不是只得到一段不可追溯的文本。
3. Agent 过程难调试：真实系统中，问题可能出在路由、检索、工具调用或记忆污染，需要可观测 Trace。
4. 面试演示需要稳定复现：项目使用 mock LLM、deterministic embedding 和稳定 mock OCR，避免依赖外部服务。

三、核心能力

EMKA 支持文本、表格和图片上传。上传后系统会自动判断资料模态，选择解析器，完成内容归一化、文档分块、embedding 生成和向量索引。用户提问时，Agent Runtime 会读取记忆、判断意图、规划任务、调用工具、执行 RAG 检索，并返回回答、报告、引用证据和执行轨迹。

四、系统架构摘要

前端使用 React、Vite、TypeScript 构建中文工作台。后端使用 FastAPI 和 Pydantic 提供 API。PostgreSQL 存储用户、会话、文档、分块、记忆、Trace 和工具调用记录。Milvus 存储文档 chunk 向量，用于 RAG 相似度检索。Agent Runtime 由 Memory Service、Intent Router、Task Planner、Tool Registry、RAG Retriever 和 Trace Recorder 组成。

五、演示亮点

1. 左侧上传资料，展示文本、表格和图像三类文档。
2. 中央对话区提问，例如“请总结我上传的资料，并引用来源”。
3. 右侧洞察区查看执行轨迹、检索文档、工具调用、记忆和报告。
4. 展示系统不仅能回答问题，还能解释回答依据和执行过程。

六、当前边界

当前项目定位为 Demo workspace，不是生产部署模板。LLM 行为是 mock，OCR 文本是 mock，但图片格式、宽高等元数据会真实读取。生产化前需要接入真实大模型、embedding provider、OCR 服务、鉴权、租户隔离、异步任务队列、监控告警和质量评估体系。
