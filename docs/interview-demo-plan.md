# EMKA 项目面试展示方案

## 1. 展示目标

本次展示的目标不是单纯跑通一个页面，而是让面试官在 8 到 12 分钟内理解三件事：

1. 我做的是一个企业多模态知识 Agent 工作区。
2. 项目覆盖了从资料上传、解析、索引、检索，到 Agent 编排、工具调用、记忆和 Trace 可观测的完整链路。
3. 我清楚当前 Demo 的工程边界，也知道如果继续生产化应该怎么扩展。

一句话介绍：

> EMKA 是一个面向企业知识场景的多模态 Agent 演示项目，用户上传文档、表格或图片后，可以通过中文工作台进行问答、摘要、报告生成，并查看检索证据、工具调用、记忆读写和完整执行轨迹。

## 2. 展示前准备

### 2.1 启动服务

推荐使用 Docker Compose 展示完整链路：

```bash
cd EMKA
docker compose up --build
```

启动后打开：

```text
http://127.0.0.1:5173
```

健康检查：

```text
http://127.0.0.1:8000/health
```

如果只是展示前端界面改版，也可以进入前端目录运行：

```bash
cd frontend
npm run dev
```

### 2.2 准备演示文件

建议准备 3 类文件，覆盖项目的多模态能力：

| 文件类型 | 示例内容 | 展示点 |
| --- | --- | --- |
| `notes.txt` 或 `README.md` | 项目说明、业务背景、会议纪要 | 文本解析、分块、摘要 |
| `metrics.csv` 或 `sales.xlsx` | 指标、销售额、成本、地区数据 | 表格解析、行范围引用 |
| `screenshot.png` 或 `diagram.jpg` | 产品截图、架构图 | 图片元数据、mock OCR、OCR 置信度 |

### 2.3 准备问题

建议提前准备 3 个问题：

```text
请总结我上传的资料，并列出最重要的三点。
```

```text
请基于知识库生成一份 Markdown 报告，并引用来源。
```

```text
这些资料里有哪些可以支持业务决策的证据？请说明来源。
```

## 3. 推荐展示流程

## 3.1 30 秒开场

可以这样说：

> 这个项目叫 EMKA，全称是 Enterprise Multimodal Knowledge Agent。它模拟的是企业内部知识助手场景：员工上传文档、表格、图片后，系统会自动完成多模态摄取、RAG 索引、Agent 工具调用、记忆读写和 Trace 记录。前端不只是聊天框，还能看到检索证据、工具调用和执行过程，方便解释系统为什么这样回答。

重点强调：

- 企业知识场景
- 多模态摄取
- RAG 检索
- Agent 编排
- 可观测 Trace

## 3.2 1 分钟介绍架构

打开项目说明或口述即可：

```text
React 工作台
  -> FastAPI API
  -> Documents API / Chat API / Trace API / Memories API
  -> 多模态摄取
  -> PostgreSQL + Milvus
  -> Agent Runtime
  -> Router / Planner / Tool Registry / Memory / Trace Recorder
```

可以这样讲：

> 前端是 React + Vite + TypeScript。后端是 FastAPI，数据层用了 PostgreSQL 存文档、分块、记忆和 Trace，向量检索使用 Milvus。聊天请求进入 Runtime 后，会先读取记忆，再做意图路由和任务规划，然后调用内部工具，例如搜索文档、读取文档、总结、生成报告，最后把回答、报告、引用证据和 Trace 一起返回。

## 3.3 2 分钟展示上传与摄取

操作步骤：

1. 打开工作台。
2. 在左侧上传一个文本文件、一个表格文件、一个图片文件。
3. 观察本地文档库出现文件，显示类型标签。
4. 指出上传结果里的切片数、索引状态。

讲解要点：

> 上传后后端会根据扩展名判断模态：文本走 Text Parser，表格走 Table Parser，图片走 OCR Parser。解析后的内容会进入 Normalizer，然后写入 documents 表，再经过 Chunker 分块、生成 deterministic embedding，并写入 Milvus。这里的 OCR 和 LLM 是 mock，是为了让 Demo 稳定可复现，不依赖外部服务。

可以主动说明边界：

> 当前 OCR 会真实读取图片格式、宽高等元数据，但文本是稳定 mock 输出；生产化时可以替换成 EasyOCR、Tesseract 或云 OCR 服务。

## 3.4 3 分钟展示问答、报告和证据

操作步骤：

1. 在底部输入框输入：

```text
请总结我上传的资料，并列出最重要的三点，同时引用来源。
```

2. 点击发送。
3. 展示中央回答。
4. 展示右侧“检索文档”和“报告”。

讲解要点：

> 这里不是只返回一段回答。后端会同时返回 retrieved_docs、tool_calls、memory_ops 和 trace_id。检索文档里会展示命中的标题、片段、相关度和引用。如果是表格数据，会保留行范围；如果是图片，会展示 OCR 置信度字段。这样做的目的是降低黑盒感，让回答能够追溯到资料来源。

## 3.5 2 分钟展示 Agent Runtime 和 Trace

操作步骤：

1. 看右侧“执行轨迹”。
2. 看“工具调用”。
3. 看“记忆”。
4. 看“操作提取”。

讲解要点：

> 聊天请求进入 Runtime 后，会经过 memory、router、planner、tools、RAG 和 trace recorder。右侧面板展示的是这条链路的可观测数据：系统判断了什么意图、置信度是多少、计划调用哪些工具、工具是否成功、耗时多少、是否读写记忆。这个设计方便开发阶段调试，也方便向用户解释 Agent 的执行过程。

可以强调工程价值：

> 我把可观测性作为一等能力，而不是只看最终 answer。因为 Agent 系统真正难调试的地方往往是路由、检索和工具调用，而不是 UI 本身。

## 3.6 1 分钟展示前端体验优化

展示点：

- 中文界面
- 左侧知识库和工具切换
- 中央沉浸式对话
- 右侧洞察区可收起
- 快捷提示词
- Enter 发送、Shift+Enter 换行

可以这样说：

> 我最近对前端做了一轮体验优化，把原来的工程调试型三栏界面改成更像产品化工作台的布局。左侧管理资料，中间专注对话，右侧保留运行证据和调试信息。这样既适合用户使用，也保留了技术展示需要的透明度。

## 4. 面试官可能追问与回答

### Q1：为什么要同时用 PostgreSQL 和 Milvus？

回答：

> PostgreSQL 适合存结构化业务数据，比如文档、分块、记忆、Trace 和工具调用；Milvus 适合做向量相似度检索。RAG 检索时先用向量在 Milvus 里找到相似 chunk，再回到 PostgreSQL 取标题、正文片段、模态、引用等业务字段。这种分层比把所有内容都塞进一个库更清晰。

### Q2：这个项目是真实调用大模型吗？

回答：

> 当前不是。这个项目定位是可稳定复现的 Demo workspace，所以 LLM 行为、embedding 和 OCR 都做了 mock 或 deterministic 实现。这样面试或本地演示不依赖外部 Key、网络和模型延迟。但架构上已经把工具、检索、编排、Trace 的接口留出来，后续可以替换真实 LLM、embedding provider 和 OCR 引擎。

### Q3：你负责了哪些部分？

回答时按自己的真实贡献调整。可以参考：

> 我主要负责项目架构设计、FastAPI API、摄取流程、RAG 检索、Agent Runtime、Trace 记录，以及前端工作台的中文化和体验改版。重点不是单个页面，而是把上传、解析、索引、检索、工具调用和可观测数据串成一条完整链路。

### Q4：如果要生产化，还缺什么？

回答：

> 主要有五类：第一，接入真实 LLM、embedding 和 OCR；第二，增加登录鉴权、租户隔离和权限控制；第三，补齐异步任务队列，避免大文件上传阻塞请求；第四，增加监控、日志、告警和成本追踪；第五，完善评测体系，比如检索命中率、引用准确率、回答质量和工具调用成功率。

### Q5：如何保证回答可信？

回答：

> 我做了两层设计。第一，RAG 返回 retrieved_docs，每条证据包含标题、片段、相关度和引用字段；第二，Trace 记录完整执行过程，包括意图、计划、工具调用、记忆读写和耗时。用户可以从回答回溯到证据，开发者可以从 Trace 定位问题。

### Q6：多模态体现在哪里？

回答：

> 上传阶段支持文本、表格和图片。文本会按文档解析，表格会保留行范围等元数据，图片会读取格式、尺寸并生成 OCR 相关字段。检索结果中也会保留 modality，所以前端可以展示文本、表格、图像不同来源的证据。

## 6. 演示时要避免的坑

1. 不要把 mock LLM 说成真实大模型能力。
2. 不要只展示 UI，要强调后端链路和数据流。
3. 不要回避当前边界，主动说明 Demo 与生产系统的差距。
4. 如果上传或服务异常，直接切到架构讲解和 API/Trace 说明，不要现场长时间排错。
5. 不要讲太多文件名，优先讲模块职责和数据流。

## 7. 推荐收尾

可以这样结束：

> 总结一下，EMKA 的价值在于它把企业知识 Agent 的关键链路做成了一个可运行 Demo：多模态摄取、RAG 检索、Agent 工具编排、记忆管理和 Trace 可观测。当前它不是生产模板，但已经具备清晰的模块边界，后续可以把 mock LLM、mock OCR 和 deterministic embedding 替换为真实服务，并补齐鉴权、异步任务和监控体系。
